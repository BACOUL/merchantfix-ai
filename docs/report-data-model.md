# MerchantFix report data model

This document defines the data model used before creating a customer-facing report page, PDFShift PDF, ZIP delivery, or email delivery.

The goal is simple:

> A MerchantFix report must be generated from structured diagnostic facts, not from vague marketing copy.

## Report principle

The report is not a Google approval promise. It is a structured Shopify product-data diagnosis based on:

- the uploaded Shopify CSV;
- the pasted Google Merchant Center warning, when provided;
- deterministic MerchantFix checks;
- guardrails that separate safe notes, manual review, and blocked automation.

## Report object

The source model is created by `buildMerchantFixReportModel` in `lib/reportDataModel.ts`.

Main fields:

| Field | Purpose |
| --- | --- |
| `reportId` | Traceable report/session identifier |
| `reportVersion` | MerchantFix report schema version |
| `methodVersion` | Diagnostic method version |
| `generatedAt` | Report generation timestamp |
| `shopName` | Optional merchant/store name |
| `csvFilename` | Uploaded CSV filename |
| `stripeSessionId` | Optional payment session identifier |
| `merchantCenterWarning` | Pasted warning text, if provided |
| `summary` | Human-readable analysis summary |
| `score` | Internal product-data readiness score, explicitly not a Google score |
| `counts` | Product, finding, severity, manual review, and blocked counters |
| `topIssues` | Most frequent issue families |
| `rowFindings` | Row-level findings with evidence and guardrail status |
| `recommendedActions` | Next actions for the merchant |
| `credibilityRules` | Rules explaining why the report is cautious |
| `limitations` | Explicit boundaries and non-promises |
| `disclaimer` | Mandatory product disclaimer |

## Product Data Readiness score

The report includes an internal score from 0 to 100.

This score is:

- generated by MerchantFix;
- based on detected CSV findings;
- not produced by Google;
- not a ranking score;
- not a guarantee of approval.

Labels:

| Label | Meaning |
| --- | --- |
| `ready` | Low detected CSV-level risk in supported checks |
| `review_needed` | Some rows need evidence before editing |
| `high_risk` | Blocked or risky rows must be reviewed before resubmission |
| `invalid_input` | The file could not be analyzed safely |

## Row finding decisions

Each finding gets a decision status.

| Status | Meaning |
| --- | --- |
| `safe_note` | MerchantFix can provide a safe note without inventing product data |
| `manual_review` | Real evidence is needed before changing Shopify data |
| `blocked` | MerchantFix should not deliver an automated correction |

## Evidence needed

Manual-review findings include evidence requirements, for example:

- product packaging barcode;
- supplier sheet;
- manufacturer product data;
- Shopify admin field;
- live storefront data;
- public image URL;
- merchant confirmation.

The report should show these requirements clearly so the merchant understands what must be verified before editing product data.

## Credibility rules

Every customer-facing report should include the following credibility principles:

1. CSV-based diagnosis only.
2. No invented product facts.
3. Manual evidence separated from safe notes.
4. Unsafe automation blocked.
5. No Google approval, ranking, traffic, sales, or account recovery guarantee.

## Future delivery flow

This model is the base for the next product layer:

```text
CSV analysis -> report data model -> HTML report page -> PDFShift PDF -> ZIP/email delivery
```

Future PRs should use this model instead of building PDFs or emails directly from raw CSV data.
